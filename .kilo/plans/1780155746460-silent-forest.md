# Fix LinkedIn Apply Login Loop

## Problem Analysis

The AI agent is stuck in a loop applying to a LinkedIn job because:

1. **LinkedIn's `apply()` method never calls `login()`** - Unlike the `search()` method which authenticates before searching (lines 126-129), the `apply()` method jumps straight to the task without checking if the user is authenticated
2. **Empty resume path** - Worker passes `resume_path or ""` at `application_worker.py:359`, which sends an empty string to the agent
3. **No loop detection** - The agent keeps retrying without recognizing it's stuck on the login modal
4. **No prerequisite validation** - The worker doesn't validate authentication or resume availability before attempting apply

## Solution

### 1. Modify LinkedIn `apply()` to authenticate before applying
**File:** `backend/app/core/automation/platforms/linkedin.py`

- Add login check at the start of `apply()` method (similar to `search()`)
- Pass credentials from environment variables (`LINKEDIN_EMAIL`, `LINKEDIN_PASSWORD`)
- If login fails, raise `AuthenticationError` with clear message instead of silent loop

### 2. Add resume path validation in LinkedIn `apply()`
- Check if `resume_path` is empty or None before starting the agent
- Raise `ApplicationSubmissionError` with clear message if no resume available

### 3. Add pre-apply validation in worker
**File:** `backend/app/workers/application_worker.py`

- Before calling `platform.apply()`, validate prerequisites:
  - Check if resume exists for the application (not just empty string)
  - Fail early with proper error message instead of sending empty path
- Handle authentication errors gracefully with meaningful messages

### 4. Improve session persistence for browser-use
**File:** `backend/app/core/automation/agent.py`

- The session manager saves cookies but they're never loaded by the browser agent
- Browser agent uses `user_data_dir` which can persist Chrome profile sessions
- Ensure session persistence works correctly by loading cookies when available
