# Plan: Application Submission (Apply Workflow) Implementation

## Overview
Connect the application creation flow to Redis queue processing, add missing platform field, and enable WebSocket updates in the UI.

---

## Step 1: Add `platform` field to `ApplicationCreate` schema
**File:** `backend/app/schemas/application.py`

Add `platform` field as a required string field (per your answer "A: Required schema"). Since Job already has `platform` field, this provides explicit control.

```python
platform: str  # Required: job platform for application submission
```

Also add to `ApplicationBatchCreate` schema with same semantics.

---

## Step 2: Enqueue applications to Redis in service layer
**File:** `backend/app/services/application.py`

Modify `create_application`:
- Accept optional `redis` parameter (or get it internally)
- After creating application, check `apply_mode`:
  - If `apply_mode == "autonomous"`: fetch Job, enqueue to Redis with full payload
  - If `apply_mode == "review"` or `"batch"`: set status to `PENDING_REVIEW`/`QUEUED` (no enqueue yet)

Modify `approve_application`:
- After updating status to `APPROVED`, fetch associated Job to get platform, then enqueue to Redis
- This replaces your answer "API + Frontend enqueue" - the backend will handle the enqueue when approve API is called

**Payload structure for worker:**
```python
{
    "job_id": str,
    "application_id": str,
    "resume_id": str | None,
    "platform": str  # From Job.platform
}
```

---

## Step 3: Fix status workflow for review mode
**File:** `backend/app/services/application.py`

In `create_application`, set status based on `apply_mode`:
- `autonomous`: Status = `QUEUED` (will be picked by worker)
- `review`: Status = `PENDING_REVIEW` (awaiting user approval)
- `batch`: Status = `QUEUED` (batch will be approved later)

## Step 5: Implement WebSocket message handling in frontend

interface UseWebSocketOptions {

Parse and route messages by type, calling the appropriate handlers.
